# Security Model

OpenClaw Auto Heal 的核心能力，是在检测到 Gateway 配置异常时，自动调用 AI 生成修复脚本，并尝试完成最小化修复。这种设计很强，但天然具备安全风险，所以必须把边界讲清楚。

## 当前安全策略

当前版本已经实现以下限制：

### 1. 修复脚本只允许操作目标配置文件
AI 生成的 Python 修复脚本，只允许读写 `openclaw.json`。

### 2. 禁止危险模块和危险调用
脚本会拦截这些能力：

- `os`
- `subprocess`
- `shutil`
- `requests`
- `urllib`
- `os.system`
- `subprocess.run/call/Popen`
- `shutil.rmtree`

### 3. 禁止网络访问
修复脚本本身不允许发起网络请求。

### 4. 禁止执行子进程
修复脚本不允许调用 shell、启动子进程、执行外部命令。

### 5. 双层校验
当前安全校验分两层：

- 字符串级危险模式拦截
- Python AST 结构级校验

## 当前仍存在的风险

虽然已经做了基础防护，但这还不是一个“高安全等级”的沙箱：

- 仍然依赖脚本规则，而不是操作系统级隔离
- 仍然依赖 Python AST 校验覆盖率
- 仍然没有真正的 seccomp / sandbox / jail 级执行隔离
- 如果 AI 请求层被污染，仍可能返回恶意但绕规则的代码

## 生产环境建议

如果你想把它用在更严肃的环境，至少建议这样做：

### 1. 使用独立修复凭证
不要直接复用主 OpenClaw 的生产模型配置。优先使用：

- `AUTO_HEAL_API_KEY`
- `AUTO_HEAL_API_ENDPOINT`
- `AUTO_HEAL_MODEL`

### 2. 保留独立安全备份
确保 `openclaw.json.safe-backup` 独立存在，且可人工恢复。

### 3. 将修复能力限制在单机、单实例
当前版本更适合个人或小团队单机部署，不建议直接用于多节点集群控制面。

### 4. 在正式启用前做故障注入测试
至少验证这些场景：

- JSON 语法损坏
- null 值错误
- 无效字段残留
- 修复脚本执行失败
- AI 返回不可解析内容

## 不适用场景

当前版本不适合：

- 高合规生产环境
- 强隔离要求的企业内控环境
- 需要多租户隔离的运行环境
- 对 AI 自动改配置零容忍的系统

## 后续安全路线

后续如果要继续往 GitHub 上更像“正式产品”推进，我建议按这个顺序加强：

1. 增加 dry-run 模式
2. 增加修复前后 diff
3. 增加允许修改字段白名单
4. 增加真正的受限执行环境
5. 增加回滚审计记录

## Responsible Use

OpenClaw Auto Heal 应被视为一个“带安全护栏的自动修复原型”，而不是万能的自治运维系统。

在没有充分测试、没有独立凭证、没有人工回滚预案之前，不建议直接把它用于高价值生产环境。
